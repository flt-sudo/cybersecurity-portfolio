Wazuh alert levels and corresponding SOC response priorities:

| Wazuh Level | Severity | SOC Priority | Expected Response Time | Action |
|---|---|---|---|---|
| 1-4 | Informational | P4 - Low | Review in daily summary | Log for trending |
| 5-7 | Low | P3 - Normal | Within 4 hours | Investigate if pattern emerges |
| 8-10 | Medium | P2 - Elevated | Within 1 hour | Investigate promptly |
| 11-13 | High | P1 - High | Within 15 minutes | Investigate immediately |
| 14-15 | Critical | P0 - Critical | Immediate | Drop everything, investigate now |

## Initial Triage Checklist

For every alert, gather the following information before beginning investigation:

### 1. Alert Metadata
- [ ] **Rule ID and description** -- What detection logic triggered?
- [ ] **Severity level** -- What is the priority?
- [ ] **Timestamp** -- When did the event occur? (Check for timezone)
- [ ] **Agent name and ID** -- Which endpoint generated the alert?
- [ ] **MITRE ATT&CK mapping** -- What technique is associated?

### 2. Source and Destination Context
- [ ] **Source IP address** -- Internal or external? Known asset?
- [ ] **Destination IP/host** -- Which system was targeted?
- [ ] **Source port / Destination port** -- Expected for the service?
- [ ] **User account** -- Which user was involved? Service account? Root?

### 3. Frequency and Pattern
- [ ] **Alert count** -- How many times has this alert fired?
- [ ] **Time range** -- Is this a burst or gradual pattern?
- [ ] **Related alerts** -- Any other alerts from the same source?
- [ ] **Historical baseline** -- Has this source triggered alerts before?

### 4. Environmental Context
- [ ] **Change window** -- Is there a scheduled maintenance or deployment?
- [ ] **Known scanning** -- Is an authorized vulnerability scan running?
- [ ] **Business context** -- Is the affected system critical?

## Investigation Procedures by Alert Type

### SSH Brute Force
**Rules: 100001-100005 | MITRE ATT&CK: T1110.001**

#### Step 1: Identify the Source
```
Dashboard path: Security Events > Filter by rule.id: 100002
```
- Note the source IP (`data.srcip` field)
- Determine if the IP is internal or external
- For external IPs, check threat intelligence:
  - AbuseIPDB: `https://www.abuseipdb.com/check/<IP>`
  - VirusTotal: `https://www.virustotal.com/gui/ip-address/<IP>`
  - Shodan: `https://www.shodan.io/host/<IP>`

#### Step 2: Assess the Scope
- How many unique target accounts are being attempted?
  - Single account = targeted attack
  - Multiple accounts = credential stuffing / spray
- What is the attempt rate? (attempts per minute)
- Are other hosts being targeted from the same IP?

#### Step 3: Check for Successful Authentication
**This is the critical step.** A brute-force attempt followed by a successful login (rule 100005) indicates compromise.

```
Dashboard filter: rule.id: 5716 AND data.srcip: <ATTACKER_IP>
```

On the endpoint, verify directly:
```bash
# Check auth.log for successful logins from the attacker IP
grep "Accepted" /var/log/auth.log | grep "<ATTACKER_IP>"

# Check last logins
last -i | grep "<ATTACKER_IP>"

# Check for current sessions from the attacker
who | grep "<ATTACKER_IP>"
```

#### Step 4: Determine Verdict
- **True Positive** -- External IP with no business justification making rapid failed login attempts
- **False Positive** -- Internal automation, misconfigured service account, or authorized penetration test
- **Benign True Positive** -- Legitimate user who forgot their password (typically 2-3 attempts, not 5+)

#### Step 5: Response
- If active and external: Block the IP via firewall (verify active response fired, or block manually)
- If successful login detected: Escalate immediately. Initiate incident response.
- Document findings in the ticketing system.

### SQL Injection
**Rules: 100100-100105 | MITRE ATT&CK: T1190**

#### Step 1: Examine the Request
- Review the full HTTP request URI from the alert data
- Identify the injection technique:
  - `UNION SELECT` -- Data extraction attempt
  - `OR 1=1` -- Authentication bypass attempt
  - `SLEEP()` -- Blind SQLi (time-based)
  - `DROP TABLE` -- Destructive intent

#### Step 2: Check the HTTP Response Code
The server response code indicates whether the injection was processed:

| Response Code | Implication |
|---|---|
| 200 | Request was processed -- may indicate vulnerability |
| 400 | Server rejected malformed request -- likely blocked |
| 403 | WAF or access control blocked the request |
| 500 | Application error -- injection likely reached the database layer |

A **200 or 500** response to an injection payload requires deeper investigation.

#### Step 3: Correlate with Application Logs
```bash
# Check Apache access log for the full request from the attacker
grep "<ATTACKER_IP>" /var/log/apache2/access.log | grep -i "union\|select\|sleep\|or 1"

# Check Apache error log for database errors
grep -i "sql\|mysql\|syntax\|query" /var/log/apache2/error.log
```

#### Step 4: Determine If Data Was Exfiltrated
- Check response sizes: Unusually large responses after SQLi payloads may indicate data extraction
- Review database logs if available
- Check for data in subsequent requests (attacker may echo results)

#### Step 5: Determine Verdict
- **True Positive** -- External IP sending injection payloads; application returned 200/500
- **False Positive** -- Security scanner (Nessus, Qualys), authorized pentest, or web crawler triggering on benign content that matches SQL keywords
- Check the User-Agent header: Tools like `sqlmap/1.x` are obvious indicators

### Cross-Site Scripting
**Rules: 100106-100108 | MITRE ATT&CK: T1190, T1059.007**

#### Step 1: Identify the Payload Type
- `<script>alert(1)</script>` -- Classic reflected XSS test
- `onerror=alert(1)` -- Event handler injection
- `javascript:alert(document.cookie)` -- Cookie theft attempt

#### Step 2: Determine Reflected vs. Stored
- **Reflected XSS:** The payload appears only in the request URI. Impact is limited to the user who clicks the crafted link.
- **Stored XSS:** The payload was submitted via a form and stored server-side. Check if subsequent requests serve the payload. Stored XSS is higher severity.

#### Step 3: Assess Impact
- Did the payload reference `document.cookie`? This is a session hijacking attempt.
- Did the payload attempt to load an external script (`src=http://...`)?
- Is the target page authenticated? XSS on login pages is lower risk than on authenticated pages.

#### Step 4: Determine Verdict
- **True Positive** -- XSS payloads in request parameters targeting a web application
- **False Positive** -- Development/testing traffic, CMS content with legitimate JavaScript, or encoded content that matches patterns

### File Integrity Monitoring
**Syscheck Alerts | MITRE ATT&CK: T1565.001**

#### Step 1: Review the Changed File
- What file was modified? (Check `syscheck.path`)
- What changed? (Check `syscheck.diff` for content changes)
- What are the before/after hashes? (`syscheck.sha256_before`, `syscheck.sha256_after`)
- What user made the change? (`syscheck.uname`)

#### Step 2: Evaluate the Context
Critical files that should rarely change outside of maintenance:

| File | Concern |
|---|---|
| `/etc/passwd` | New user added, UID 0 escalation |
| `/etc/shadow` | Password changed |
| `/etc/sudoers` | Privilege escalation |
| `/etc/ssh/sshd_config` | SSH hardening weakened |
| `/etc/crontab`, `/etc/cron.d/*` | Persistence mechanism |
| `/var/www/html/*` | Web shell deployment, defacement |
| `/root/.ssh/authorized_keys` | Unauthorized SSH key added |

#### Step 3: Correlate with System Activity
```bash
# Who was logged in at the time of modification?
last -t $(date -d "2025-03-05 14:23:00" +%Y%m%d%H%M%S)

# What processes ran around that time? (check audit log)
ausearch -ts "03/05/2025" -te "03/05/2025" -f /etc/passwd

# Check sudo log for privilege escalation
grep "sudo" /var/log/auth.log | grep "Mar  5 14:2"
```

#### Step 4: Determine Verdict
- **True Positive** -- Unauthorized modification (no change ticket, no known admin activity)
- **False Positive** -- Package update (apt upgrade), configuration management (Ansible/Puppet), scheduled maintenance

### Suspicious Process Execution
**MITRE ATT&CK: T1059.004**

#### Step 1: Identify the Process
- Process name, PID, parent process (PPID)
- Command line arguments
- User context (root? service account? regular user?)
- Working directory

#### Step 2: Evaluate Suspicion Indicators
High-confidence indicators of malicious activity:

- Processes running from `/tmp`, `/dev/shm`, or user home directories
- Base64-encoded command line arguments (`echo <base64> | base64 -d | bash`)
- Netcat (`nc`, `ncat`) with `-e` or `-c` flags (reverse shell)
- `curl` or `wget` piped to `bash` or `sh`
- Python/Perl one-liner reverse shells
- Processes spawned by web server user (`www-data`)

#### Step 3: Investigate the Process Tree
```bash
# Check if the process is still running
ps aux | grep <PID>

# Examine the process tree
pstree -p <PID>

# Check open files and network connections
ls -la /proc/<PID>/fd/ 2>/dev/null
cat /proc/<PID>/cmdline 2>/dev/null | tr '\0' ' '

# Check network connections for the process
ss -tnp | grep <PID>
```

#### Step 4: Determine Verdict
- **True Positive** -- Process with no legitimate business purpose, running in suspicious context
- **False Positive** -- Authorized admin activity, scheduled maintenance script, monitoring agent

## False Positive Identification

Common false positive scenarios and how to identify them:

| Scenario | How to Identify | Tuning Action |
|---|---|---|
| Authorized vulnerability scan | Source IP is scanner (Nessus, Qualys); scheduled scan window | Whitelist scanner IP in rule exceptions |
| Automated service account | Failed auth from known service; same pattern daily | Create exception for service account + source IP |
| Package manager updates | FIM alerts during apt/yum update window | Add `<ignore>` for package-managed paths or tune frequency |
| CMS content with JS | XSS rule matches legitimate blog post content | Add exception for content management paths |
| Developer testing | Internal IP, during business hours, test environment | Tag test environments to reduce alert priority |
| Monitoring agents | Repeated command execution from nagios/zabbix user | Whitelist monitoring user in rule |

### False Positive Documentation

When identifying a false positive, document:
1. The rule ID and alert details
2. Why it is a false positive (evidence)
3. The recommended tuning action
4. Whether the tuning has been implemented

## Escalation Criteria

Immediately escalate to the security team lead / incident response when:

- [ ] **Confirmed successful compromise** -- Successful login after brute-force (rule 100005)
- [ ] **Data exfiltration indicators** -- Large HTTP responses after SQLi; outbound connections to unknown IPs
- [ ] **Unauthorized privilege escalation** -- Changes to /etc/sudoers, new UID 0 accounts
- [ ] **Web shell detected** -- New PHP/JSP/ASP files in web directories with execution permissions
- [ ] **Persistence mechanisms** -- New cron jobs, SSH keys, systemd services without change tickets
- [ ] **Active attacker on the network** -- Multiple hosts targeted, lateral movement indicators
- [ ] **Ransomware indicators** -- Mass file encryption, ransom notes, MBR modification
- [ ] **Critical system level 14-15 alerts** -- Any alert at critical severity

## Response Actions

### Containment Actions (by severity)

| Severity | Action | Authority Required |
|---|---|---|
| Medium (8-10) | Monitor, add to watchlist | SOC Analyst |
| High (11-13) | Block source IP, disable compromised account | SOC Analyst + Team Lead approval |
| Critical (14-15) | Isolate host from network, block at perimeter | Incident Commander approval |

### Common Response Commands

```bash
# Block an IP address via iptables
sudo iptables -I INPUT -s <ATTACKER_IP> -j DROP

# Disable a compromised user account
sudo usermod -L <USERNAME>
sudo pkill -u <USERNAME>

# Kill a malicious process
sudo kill -9 <PID>

# Quarantine a suspicious file (preserve for forensics)
sudo mv /path/to/file /var/quarantine/
sudo chmod 000 /var/quarantine/file

# Capture network traffic for forensics
sudo tcpdump -i eth0 host <ATTACKER_IP> -w /tmp/capture.pcap
```
